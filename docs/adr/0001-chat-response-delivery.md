# ADR-0001: チャット応答の配信方式

- **ステータス**: 承認済み — 案2（SSE手組み）を採用し Step 6.5 として実装
- **日付**: 2026-07-07（提案・承認同日）
- **決定者**: naoto（技術判断） / Claude（選択肢・トレードオフ整理）

## 文脈

`POST /api/chat` は現在、全文生成の完了を待って JSON を1回返す（一括レスポンス）。実測で送信から表示まで約2.4秒の「沈黙」があり、チャットUIとしての品質感を大きく損ねている。体感改善のための配信方式を選定する。

前提となる制約・特性：

- 生成は OpenAI Chat Completions（`gpt-4o-mini`）。SDK は `stream: true` でチャンク列を返せる
- 回答本文（トークン列）とは別に、**出典リストという構造データ**を必ずクライアントに届ける必要がある（このアプリの生命線）
- 出典は retrieve 完了時点（生成開始前）に確定している
- ホスティングは未定。常駐プロセス型かサーバーレスかで実現可能な方式が変わる
- 学習プロジェクトであり、「仕組みが見えること」自体に価値がある

## 決定要因（判断基準）

1. **学習効果** — 中身がブラックボックスにならないこと（本プロジェクトの主目的）
2. **体感レイテンシ** — 最初の文字が出るまでの時間
3. **工数** — 実装＋異常系の動作確認まで
4. **本番適合性** — 実務でお客さんに提案するときの筋の良さ
5. **デプロイ制約** — サーバーレス環境でも成立するか

## 選択肢

### 案1: 一括レスポンス（現状維持）

生成完了後に `Response.json({answer, sources, usage})` を1回返す。

- ✅ 工数ゼロ。エラー処理が HTTP ステータス1系統で完結し、最もシンプル
- ✅ どんなホスティングでも動く
- ❌ 2.4秒の沈黙はそのまま。回答が長くなるほど悪化する（沈黙時間は回答トークン数に比例）
- 工数: **0**

### 案2: SSE 手組み（`ReadableStream` + `text/event-stream`）

`generate.ts` に `stream: true`、Route が `type` 付き JSON イベント（`sources` / `token` / `done` / `error`）を SSE で流し、UI がリーダーで逐次描画する。詳細設計は図解（rag-study01-sse-design-change.html）参照。

- ✅ 最初のトークンから描画開始（沈黙 約2.4秒 → 約0.5秒）
- ✅ 外部依存ゼロ。Web標準（SSE / ReadableStream）だけで完結し、**ストリーミングの仕組みが全部見える**
- ✅ 出典の多重化を自分でイベント設計する経験が得られる
- ❌ エラーが「HTTPステータス」と「errorイベント」の2系統に分裂。異常系の確認項目が増える
- ❌ サーバーレスでは関数の実行時間上限に注意（生成が長いと切られる）
- 工数: **半日弱**（うち異常系確認が1/3）

### 案3: 生HTTPチャンクストリーミング（`text/plain` 逐次書き込み）

SSE のイベント形式を使わず、本文テキストをそのまま `ReadableStream` に書き流す。出典はレスポンスヘッダ（または先頭行の埋め込みJSON）で渡す。

- ✅ SSE よりさらに実装が薄い（イベントのパース不要、UI は受けた文字を足すだけ）
- ✅ 体感改善は案2と同等
- ❌ **出典・usage・エラーを渡す場所がない**のが致命的。ヘッダに日本語ファイル名を載せるにはエンコードが必要、途中エラーは本文に混ざる、と構造データの扱いが全部ハックになる
- ❌ 「動くが汚い」を学ぶには良いが、拡張（イベント種別の追加）の余地がない
- 工数: **2〜3時間**（ただし出典まわりの負債が残る）

### 案4: WebSocket

双方向の常時接続を張り、サーバーがトークンを push する。

- ✅ 双方向なので生成の途中キャンセル・タイピング通知など将来のリッチ化に強い
- ❌ **Next.js の Route Handler は WebSocket を扱えない**。カスタムサーバー化するか、別プロセス（socket.io サーバー等）を立てる必要があり、構成が一変する
- ❌ サーバーレスとほぼ非互換。ロードバランサ・再接続の運用も発生
- ❌ このアプリの通信は「質問→回答」の単方向で、双方向性への投資が回収できない
- 工数: **1〜2日**＋インフラ構成の変更

### 案5: 非同期ジョブ＋ポーリング

`POST /api/chat` はジョブIDを即返し、生成はバックグラウンドで進める。クライアントは `GET /api/chat/:id` を定期ポーリングし、途中経過（生成済みテキスト）と完了を取得する。

- ✅ **サーバーレスと最も相性が良い**（各リクエストが短命）。リロードしても途中から再開できる
- ✅ 生成に数分かかる重い処理（長文要約・バッチ）への正攻法で、実務の引き出しとして価値が高い
- ❌ ジョブ状態の保存先が必要（現構成ならメモリ→また揮発性問題、実質 Redis/DB が要る）
- ❌ 体感はポーリング間隔（例: 500ms）で刻まれ、ストリーミングよりカクつく。リクエスト数も増える
- ❌ この規模のチャットには明らかに過剰
- 工数: **1日**＋状態ストアの選定

### 案6: Vercel AI SDK（`ai` パッケージ）採用

`streamText()` + `useChat()` フックに乗り換える。ストリーミング配信・イベント多重化・UI の逐次更新を SDK が全部やる。

- ✅ 実装が最速級で、途中中断・再送・ツール呼び出しなど本番機能が揃っている。**実務ならまずこれを提案する**
- ✅ プロトコル（SSEベースの data stream）は SDK が管理し、バグの入る余地が少ない
- ❌ まさにその理由で**学習効果が下がる**——このADRで学びたい多重化・エラー2系統問題が SDK の内側に隠れる
- ❌ OpenAI 呼び出しが `@ai-sdk/openai` 経由になり、Step 1 で作った `lib/openai.ts` の直呼び構成と二重化する
- ❌ 依存とバージョン追従が増える（学習リポジトリの寿命を考えると負債）
- 工数: **半日**（既存構成の置き換え含む）

### 検討したが除外

- **gRPC ストリーミング**: ブラウザ直は grpc-web + プロキシが必要で、この規模では論外
- **RSC / Server Actions のストリーミング**（`createStreamableValue` 等）: フォームアクションと密結合になり、API として独立させた現構成（curl でも叩ける）を壊す

## 推奨

**案2（SSE手組み）を推奨する。**

- 本プロジェクトの主目的は学習であり、「トークン列と構造データの多重化」「エラーの2系統分裂」というストリーミングの本質的な難所を、依存ゼロで一度手で通しておく価値が大きい
- 一度手組みを経験していれば、実務で案6（AI SDK）を採用するときに「SDKが何を隠蔽しているか」を説明できる——見積もり訓練の観点で最も投資対効果が高い
- 案5（ジョブ型）は「サーバーレス×重い生成」の実務パターンとして頭に置いておき、ここでは実装しない

## 帰結（採用した場合）

- `generate.ts` / `api/chat/route.ts` / `page.tsx` の3ファイルが変わる（`VectorStore` / `retrieve` / `embed` は無傷）
- イベント契約（`sources` → `token`* → `done` | `error`）がクライアントとの新しいAPI仕様になる。ADR-0002 以降で API を清書する場合はこの契約を含めること
- 異常系（途中切断・APIエラー・連打）の動作確認が新たなテスト項目として増える
- 将来サーバーレスにデプロイする場合は、関数実行時間の上限と SSE の相性を再評価すること（このADRの見直しトリガー）
